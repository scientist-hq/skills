## Safety Checks (do these first, stop if any fail)

1. **Not on main**: Run `git branch --show-current` â€” if on `main` or `staging`, STOP and tell the user.
2. **Clean working tree**: Run `git status` â€” if there are uncommitted changes, STOP and ask the user to commit or stash first.
3. **Pushed to remote**: Run `git log origin/main..HEAD --oneline` to see commits. If the branch isn't pushed, run `git push -u origin HEAD`.

## Gather Context

1. **Read the commits**: Run `git log origin/main..HEAD --oneline` and `git diff origin/main...HEAD --stat` to understand what changed.
2. **Find the plan**: Check `plans/` for a plan file matching this branch name â€” read it for context.
3. **Find the issue**: Look for a GitHub issue number in:
   - The branch name (e.g., `34869-legal-amendment-text-color`)
   - The plan file
   - The user's `$ARGUMENTS`
   - If none found, ASK the user for the issue number.

## Determine Labels

Choose labels based on the type of change. Use EXACTLY these label names:

| Change Type | Label |
|------------|-------|
| New functionality | `Type: Feature` |
| Enhancement to existing feature | `Type: Improvement` |
| Bug fix | `Type: Fix` |
| Code cleanup / refactoring | `Type: Cleanup` |
| Infrastructure / tooling | `Type: Infrastructure` |
| Urgent production fix | `Hotfix` |
| Database migration included | `Migration` |
| UI/design changes | `Style` |
| Accessibility improvements | `Accessibility` |
| Dependency updates | `Dependencies` |
| Internal (exclude from release notes) | `Internal` |

Also add area labels if applicable:
- `Storefront` â€” client-facing changes
- `Backoffice` â€” supplier-facing changes
- `Accounting` â€” financial/billing changes
- `Security` â€” security-related changes

If unsure about labels, ASK the user.

## Create the PR

```bash
gh pr create \
  --title "<concise title under 70 chars>" \
  --body "$(cat <<'EOF'
<body content>
EOF
)" \
  --label "Type: Feature" \
  --label "Storefront" \
  --draft
```

- Always create as **draft** unless the user says otherwise
- Keep the title short and descriptive (under 70 chars)
- Use `Fixes #N` (not `Closes`) to reference the issue
- Add multiple `--label` flags as needed

## After Creation

1. Print the PR URL
2. Summarize: title, labels, linked issue, draft status
3. Remind: "Mark as ready for review when you're satisfied: `gh pr ready`"
