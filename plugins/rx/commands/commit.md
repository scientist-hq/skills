You are creating clean, well-organized git commits from the current working tree changes. Your goal is to break changes into small, logical, bite-sized commits that each address a single concern.

## Safety Checks

1. **Not on main**: Run `git branch --show-current` — if on `main` or `staging`, STOP.
2. **Has changes**: Run `git status --short` — if nothing to commit, STOP and say so.

## Workflow

1. **Analyze changes**: Run `git diff --stat` and `git diff` to understand ALL current changes.
2. **Group by concern**: Organize changes into logical commits. Each commit should be one of:
   - A single migration
   - A model change and its spec
   - A service and its spec
   - A controller change and its spec
   - A view change
   - A config/infrastructure change
   - A refactor (separate from new functionality)

3. **Order commits logically**: Dependencies first.
   - Migrations before models
   - Models before services
   - Services before controllers
   - Controllers before views
   - Specs alongside the code they test (same commit)

4. **Stage and commit each group**:
   ```bash
   git add <specific files>
   git commit -m "<message>"
   ```

5. **After all commits**: Run `git log --oneline origin/main..HEAD` and show the result.

## Commit Message Style

Follow the team's existing style (short, lowercase-start, descriptive):

```
Add supplier approval notification service
Update quote group model with approval state machine
Fix partial provider credit calculation
Remove deprecated shipping rate lookup
Change legal amendment text color for readability
```

### Rules

- **No AI attribution** — never mention Claude, AI, copilot, or "generated"
- **No conventional commits** prefix (no `feat:`, `fix:`, `chore:`) unless the team uses them
- **Imperative mood** — "Add feature" not "Added feature" or "Adds feature"
- **Under 72 characters**
- **No period** at the end
- **Describe the what** — the diff shows the how, the PR describes the why

### Good Examples
```
Add email notification for SOW acceptance
Update search index to include supplier region
Fix N+1 query in quote groups listing
Remove unused Paperclip attachment from Document model
Add RSpec tests for legal name service edge cases
```

### Bad Examples
```
Update files                          # too vague
Fix stuff                             # meaningless
Generated by Claude                   # NO
feat: add notification service        # wrong style for this team
Add email notification for SOW acceptance.  # no period
```

## Grouping Strategy

When analyzing the diff, think about what a reviewer would want to see as a single reviewable unit:

**One commit together:**
- `app/services/foo_service.rb` + `spec/services/foo_service_spec.rb`
- `app/models/bar.rb` + `spec/models/bar_spec.rb`
- `app/controllers/baz_controller.rb` + `app/views/baz/index.html.haml`

**Separate commits:**
- A migration (always its own commit)
- A refactor vs. new functionality in the same file
- Unrelated config changes

## Communication

- Show the planned commit breakdown BEFORE executing — list each commit with its files and proposed message
- ASK the user to confirm before committing
- After committing, show the final `git log --oneline` summary
